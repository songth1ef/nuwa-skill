# 人机分工评估基线

> 此基线由用户在 2026-04 探索 Claude Code 作为 harness 的过程中沉淀。
> 用于每次 AI 编码工具升级时的对照评估。

---

## 10 项人类能力基线

每次升级时，对每项判断：被替代 🟢 / 被增强 🟡 / 仍需人做 🔴 / 新增 ⚫

| # | 能力 | 含义 | 为什么重要 |
|---|------|------|----------|
| 1 | 会提问 | 把需求清晰描述给 AI，拿到基本可用代码 | AI 输出质量上限由提问质量决定 |
| 2 | 会验证 | 判断 AI 输出是否正确，能跑通、能测试 | **杠杆最大**：决定单次产出能否收敛 |
| 3 | 会拆解 | 把复杂任务拆成 AI 能处理的小步骤 | 上下文窗口与 agent 失败率的硬约束 |
| 4 | 会约束 | 写好 CLAUDE.md / 规范文件，让 Agent 持续输出一致代码 | **杠杆最大**：决定长期不退化 |
| 5 | 会架构 | 设计模块边界和数据流，AI 只填实现，你定骨架 | AI 不擅长跨模块决策 |
| 6 | 会编排 | 同时调度多个 Agent 并行工作，合并结果，解决冲突 | 多 agent 时代的核心新增能力 |
| 7 | 会判断 | AI 给出多个方案时，基于业务/性能/维护性做出取舍 | AI 不知道你的业务约束 |
| 8 | 会造工具 | 为 Agent 写 MCP Server / Skill / Hook，扩展能力边界 | 通用工具无法覆盖领域专用需求 |
| 9 | 会设计系统 | 定义整个 AI 辅助开发工作流，让团队人+Agent 协作效率最大化 | 团队规模化的瓶颈 |
| 10 | 会定义问题 | 发现别人没看到的问题和机会，AI 是你意志的延伸，你决定造什么 | **永远不会被替代** |

### 关键观察
- **第 2、4 项是杠杆最大的**：验证决定收敛，约束决定长期质量
- **第 10 项永远不会被替代**：定义"该造什么"是纯人类工作
- **第 6、8 项是新增杠杆**：随着多 agent / 工具生态成熟，越来越重要

## 5 个核心协作文件

每次升级时，对每个文件判断：是否需要更新？

| # | 文件 | 内容 | 升级时常见更新点 |
|---|------|------|---------------|
| 1 | `CLAUDE.md` | Agent 行为指令（角色、禁止项、输出规则，入口文件指向其他文件） | 新模型的输出风格变化、新工具的禁用清单 |
| 2 | `conventions.md` | 编码规范（命名、目录结构、组件模式、Git 规范） | 新工具能自动处理的规范可以放手 |
| 3 | `domain.md` | 业务描述（核心概念、术语表、业务规则） | 通常稳定，除非业务本身变 |
| 4 | `architecture.md` | 系统架构（模块边界、数据流、技术选型理由） | 新工具开启的新架构可能性（如 background agent） |
| 5 | `lessons.md` | 教训记忆（踩过的坑、已验证的决策、不要重复犯的错误） | 旧坑可能因新工具而消失，需清理 |

### 关键观察
- `CLAUDE.md` 是**入口文件**，应保持薄，按需指向其他文件
- `lessons.md` 需要**定期清理**，否则会累积过时教训误导新工具
- 90% 其他能力 Claude Code 内置了，不要重复造

## 评估时的反例提醒

- ❌ 不要被 release notes 的营销话术带偏，只看真实能力
- ❌ 不要假设"全部能力都被增强了"，多数升级只动 1-2 项
- ❌ 不要忽略"仍需人做"的项——这些恰恰是你的护城河
- ❌ 不要在没跑过真实任务前下结论
